# Plan: full docs parity for Fakturoid SDK

## Goal

Bring the SDK to full parity with `docs/`, not only for endpoint reachability, but also for documented CRUD/action/download flows, request and response shapes, changelog additions, and documented runtime behavior.

## Recommended approach

1. Build a docs-to-SDK coverage matrix first.
   - Audit `docs/authorization.md`, `docs/account.md`, `docs/users.md`, `docs/bank-accounts.md`, `docs/number-formats.md`, `docs/subjects.md`, `docs/invoices.md`, `docs/invoice-payments.md`, `docs/invoice-messages.md`, `docs/expenses.md`, `docs/expense-payments.md`, `docs/inbox-files.md`, `docs/inventory-items.md`, `docs/inventory-moves.md`, `docs/generators.md`, `docs/recurring-generators.md`, `docs/events.md`, `docs/todos.md`, `docs/webhooks.md`, and `docs/changelog.md`.
   - For every documented section, map: repository method, request params/path, response model, export status, request parity coverage, model parity coverage, and live-test coverage.
   - Treat this matrix as the source of truth for the rest of the work so gaps are explicit before code changes begin.

2. Close API surface gaps in repositories and public exports.
   - Review all files under `lib/src/repositories/*_repository.dart` against the coverage matrix and adjust signatures, query params, return types, and action/download helpers where docs and SDK diverge.
   - Prioritize already visible mismatches first, especially `lib/src/repositories/webhooks_repository.dart`, where failed deliveries currently take `page` but do not expose a paginated return type like the rest of the SDK.
   - If new public models or helpers are needed, wire them through `lib/src/fakturoid_client.dart` and `lib/fakturoid_api.dart`.

3. Close model, enum, and runtime-behavior gaps.
   - Audit `lib/src/models/**/*.dart` against docs attribute tables and changelog entries, including nested payloads, read-only/write-only behavior, serialization rules, and any newly documented fields.
   - Review webhook failed-delivery and delivery payload support so the SDK exposes all documented fields, not just a partial subset.
   - Review `lib/src/core/exceptions/fakturoid_error_interceptor.dart` and adjacent exception/response infrastructure for documented runtime behavior, especially anything derived from changelog items such as rate-limit headers or additional response metadata.
   - If response metadata must become public, extend `lib/src/core/responses/paginated_response.dart` or add a sibling response-meta type without breaking existing repository ergonomics more than necessary.

4. Strengthen automated parity tests so docs drift becomes detectable.
   - Expand `test/request_parity_test.dart` so every documented endpoint/action/download/search/pagination path is asserted for method, path, query, payload, and expected response shape.
   - Expand `test/model_and_error_parity_test.dart` so every documented/changelog field or runtime behavior that matters to consumers has a direct test.
   - Add a dedicated docs coverage test if needed (for example `test/docs_coverage_test.dart`) so each doc section must have a mapped SDK surface and at least one automated assertion.

5. Expand live coverage to representative full workflows.
   - Audit `test/live_*.dart` and ensure each API area exercises the important read/create/update/delete/action/download flow for that domain, not just a list or smoke call.
   - Focus first on invoices, expenses, inventory items, inventory moves, recurring generators, inbox files, and webhooks because they carry the most specialized behavior.
   - Keep fixture cleanup centralized through `test/live_cleanup_test.dart` and shared helpers in `test/support/`.

## Critical files likely to change

- `lib/src/repositories/*_repository.dart`
- `lib/src/models/**/*.dart`
- `lib/src/core/responses/paginated_response.dart`
- `lib/src/core/exceptions/fakturoid_error_interceptor.dart`
- `lib/src/fakturoid_client.dart`
- `lib/fakturoid_api.dart`
- `test/request_parity_test.dart`
- `test/model_and_error_parity_test.dart`
- `test/live_*.dart`
- `test/support/live_test_support.dart`
- optionally a new `test/docs_coverage_test.dart`

## Verification

1. Run `dart format .`.
2. Run `dart analyze`.
3. Run `flutter test`.
4. Run targeted live suites against the isolated tenant, for example with `FAKTUROID_SLUG=testcompany1 flutter test test/live_webhooks_test.dart`, then repeat for the touched domains.
5. Confirm the final coverage matrix has no unmapped documented endpoint, action, download flow, model field, or changelog item left unsupported.
