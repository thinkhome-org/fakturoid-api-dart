# Plán: endpoint tabulka + integrační cheat-sheet + Dart klient návrh

## Cíl
- Navázat na kompletní nastudování `docs/` a dodat tři výstupy v požadovaném pořadí: `3)` úplnou endpoint tabulku, `1)` praktický integrační cheat-sheet, `2)` návrh typů a API klienta v **Dartu**.
- Přizpůsobit výstup existující architektuře tohoto repozitáře místo původně navrhovaného TypeScriptu.

## Zmapovaný rozsah
- Celkem `21` souborů v `docs/`.
- Základ API: `docs/introduction.md`, `docs/authorization.md`, `docs/changelog.md`.
- Účet a reference: `docs/account.md`, `docs/users.md`, `docs/bank-accounts.md`, `docs/number-formats.md`.
- Fakturace a výdaje: `docs/invoices.md`, `docs/invoice-payments.md`, `docs/invoice-messages.md`, `docs/expenses.md`, `docs/expense-payments.md`, `docs/subjects.md`, `docs/inbox-files.md`.
- Automatizace a sklad: `docs/generators.md`, `docs/recurring-generators.md`, `docs/inventory-items.md`, `docs/inventory-moves.md`.
- Audit a integrace: `docs/events.md`, `docs/todos.md`, `docs/webhooks.md`.

## Doporučený postup
1. Syntetizovat společná pravidla API: base URL, OAuth flow, `User-Agent`, JSON payloady, stránkování, rate limiting, chybové formáty a account slug workflow.
2. Zpracovat doménový přehled po blocích: účet/reference, fakturace, výdaje, inbox/OCR, generátory, sklad, audit/eventy/webhooky.
3. U každé domény vypsat endpointy, klíčové atributy, stavové přechody, oddělené write-only/read-only fieldy a návaznosti mezi zdroji.
4. Explicitně zachytit cross-cutting workflow: subject -> invoice/expense snapshot, payments jako separátní endpointy, PDF/attachment downloady, inventory item -> inventory move -> line linkage, webhook/event/todo rozdíly.
5. Na závěr dodat seznam integračních rizik a nejasností v dokumentaci, zejména nekonzistentní enumy a rozdílné chování chybových stavů.

## Repo kontext pro Dart variantu
- Projekt je Dart/Flutter balíček (`pubspec.yaml`) s `dio`, `freezed`, `json_serializable`, `flutter_secure_storage`.
- Veřejný vstupní bod je `lib/fakturoid_api.dart`.
- Hlavní orchestrátor je `lib/src/fakturoid_client.dart`; public API je ve tvaru `client.<repository>.<method>()`.
- OAuth a token lifecycle řeší `lib/src/auth/auth_repository.dart` + `TokenStorage`.
- Repozitáře už existují pro většinu domén z `docs/`: account, users, subjects, invoices, invoice payments/messages, expenses, expense payments, bank accounts, number formats, inventory items/moves, generators, recurring generators, inbox files, todos, events, webhooks.
- Testy v `test/fakturoid_api_test.dart` už ověřují request/response paritu vůči dokumentaci a budou sloužit jako referenční konvence pro případné budoucí SDK změny.

## Finální výstup po schválení
- Dodat tři navazující výstupy v češtině a v tomto pořadí: endpoint tabulka -> integrační cheat-sheet -> Dart klient/type návrh.
- Endpoint tabulka bude mapovat `docs/` endpointy na existující Dart repozitáře a metody, případně je explicitně označí jako chybějící nebo problematické.
- Cheat-sheet bude psaný pro použití přes `FakturoidClient` a zohlední auth, account slug bootstrap, payments, downloads, retry a webhook idempotenci.
- Dart část nebude navrhovat generický SDK od nuly, ale naváže na současný tvar `lib/fakturoid_api.dart`, `FakturoidClient`, `freezed` modely a tenké repository wrappery.

## Doporučený postup pro realizaci výstupů
1. Sestavit úplnou endpoint tabulku po doménách a u každého endpointu doplnit: HTTP metodu, cestu, hlavní query/body parametry, návratový typ/resource, stavové zvláštnosti a mapování na existující Dart repository metodu.
2. Z této tabulky odvodit praktický integrační cheat-sheet v Dartu: přihlášení, bootstrap přes `/user.json`, práce s dokumenty, payments, attachmenty, OCR, inventory, events/todos/webhooks.
3. Nakonec dodat Dart návrh typů a klienta jako architektonické zhodnocení aktuálního balíčku: co už sedí dobře, jaké enum/model patterny držet, kde jsou ergonomické hrany a co případně rozšířit dál.

## Kritické poznatky, které musí být ve výsledku
- `docs/introduction.md` + `docs/authorization.md`: OAuth 2, povinný `User-Agent`, paginace po 40, `429` rate limiting, `402` při blokovaném účtu, dva tvary chybových payloadů.
- `docs/invoices.md` a `docs/expenses.md`: největší a nejdůležitější zdroje; obsahují CRUD, search, attachmenty, line items, locking a výpočetní pole.
- `docs/invoice-payments.md` a `docs/expense-payments.md`: platby jsou samostatná resource vrstva mimo CRUD dokumentů.
- `docs/generators.md` a `docs/recurring-generators.md`: v API v3 jsou oddělené; recurring má pause/activate a plánovací logiku.
- `docs/inventory-items.md` a `docs/inventory-moves.md`: množství položky se po vytvoření mění přes moves, ne přes update itemu.
- `docs/webhooks.md`, `docs/events.md`, `docs/todos.md`: tři různé integrační/audit kanály s odlišným účelem; webhooky navíc vyžadují idempotentní zpracování přes `Idempotency-Key`.
- `docs/changelog.md`: obsahuje důležité breaking changes a nové atributy, které nemusí být zřejmé jen z jednotlivých resource dokumentů.
- `lib/src/fakturoid_client.dart`: už dnes obsahuje account-scoped `Dio`, auth/error interceptory a registry repozitářů; výsledný Dart návrh musí tenhle shape respektovat.
- `test/fakturoid_api_test.dart`: existuje parity test styl vůči docs; pokud budu doporučovat budoucí SDK rozšíření, budu je navrhovat v tomto testovacím stylu.

## Známé nejasnosti / místa pro zvýšenou pozornost
- V dokumentaci se místy rozcházejí enumy a příklady, např. `vat_price_mode` u faktur.
- U některých delete/error scénářů jsou v textech a příkladech rozdílné status kódy (`403` vs `422`).
- Některé ukázky odpovědí používají `attachments: null`, jiné `attachments: []`.
- U recurring generatorů jsou v některých URL příkladech pravděpodobně pozůstatky cesty pro běžné generátory.
