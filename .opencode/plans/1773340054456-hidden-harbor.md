## Cíl

Prestavet live test vrstvu tak, aby `docs/` bylo mapovane skoro 1:1 na samostatne Dart test soubory, SDK bylo overene proti dedikovanemu tenantovi `testcompany1` (`TestCompany`) pro vsechny docs-backed funkce a endpointy, a hlavni live suite v tenantovi realne vytvarela a upravovala co nejvic dat bez automatickeho mazani na konci behu.

## Doporuceny pristup

1. Udelat finalni matici `docs sekce -> SDK metoda -> samostatny test file` a zkontrolovat, ze kazdy docs-backed endpoint ma sve misto.
2. Rozdelit live testy na soubory po docs sekcich, typicky:
   - `test/live_authorization_test.dart`
   - `test/live_account_test.dart`
   - `test/live_users_test.dart`
   - `test/live_bank_accounts_test.dart`
   - `test/live_number_formats_test.dart`
   - `test/live_subjects_test.dart`
   - `test/live_invoices_test.dart`
   - `test/live_invoice_payments_test.dart`
   - `test/live_invoice_messages_test.dart`
   - `test/live_expenses_test.dart`
   - `test/live_expense_payments_test.dart`
   - `test/live_inbox_files_test.dart`
   - `test/live_inventory_items_test.dart`
   - `test/live_inventory_moves_test.dart`
   - `test/live_generators_test.dart`
   - `test/live_recurring_generators_test.dart`
   - `test/live_todos_test.dart`
   - `test/live_events_test.dart`
   - `test/live_webhooks_test.dart`
   - plus `test/live_cleanup_test.dart` jako separatni destructive cleanup suite
3. Zachovat shared helper vrstvu pro bootstrap klienta, fixture tvorbu a lookup ID zavislosti, ale samotne assertions a create/read/update flow drzet v souboru konkretni docs sekce.
4. Pro writeable sekce v prazdnem tenantovi nejdriv vytvorit vsechny potrebne zavislosti a potom v ramci konkretniho test file overit create -> read/detail/list/search -> update -> action/download, bez implicitniho spolehuti na cizi data.
5. Pro delete endpointy mit separatni `live_cleanup_test.dart`, ktery explicitne vola vsechny destructive SDK delete/cleanup funkce a overi, ze po cleanupu nezustaly OpenCode fixture.
6. Read-only nebo action-only sekce (`account`, `users`, `bank-accounts`, `number-formats`, `events`, cast `todos`, cast `authorization`) testovat vlastnim souborem pouze v rozsahu, ktery docs realne podporuji.
7. Hlavni create/edit live suite nastavit tak, aby v `testcompany1` ponechavala vytvorena data; cleanup zachovat jako separatni, volitelne spousteny test.

## Aktualni uzivatelske upresneni

- Uživatel explicitne chce: `kazda funkce/kazda sekce z docs/ ma vlastni Dart test file`.
- Uživatel nechce prijmout pouze read/action coverage u read-only sekci; chce `vynutit CRUD vsude`, tj. aby test soubory pracovaly s realne vytvorenymi prerequisites z jinych sekci a maximalne simulovaly create/edit/read tok i tam, kde samotna docs sekce ma jen read nebo action endpointy.
- Prakticka interpretace pro implementaci: pro read-only sekce vytvorit potrebne fixture pres jine sekce v `setUp` / shared fixture graph a vlastni sekcni file pouzit k overeni jejich list/detail/action API. Skutecne delete endpointy zustanou centralizovane v separatnim cleanup testu.
- Nejnovější upresneni: hlavni live beh ma v tenantovi `testcompany1` realne vytvorit a upravit co nejvice dat a tato data po behu ponechat; cleanup ma zustat oddeleny a spoustet se jen explicitne.

## Implementacni faze

### Faze 1 - Audit mapy pokryti
- Sepsat checklist vsech endpointu/funkci z docs.
- Oznacit kazdy endpoint jako: implementovany / castecne implementovany / chybi / nejasny.
- U changelogu udelat samostatny checklist novych fieldu a behavioru.

### Faze 2 - Opravy SDK surface
- Dodelat nebo sjednotit repository metody, payload mapping, return types a helpery.
- Dodelat chybejici model fields / enum values / serialization nuance.
- Zkontrolovat exporty v `lib/fakturoid_api.dart` a README priklady.

### Faze 3 - Test coverage
- Rozdelit live testy po docs sekcich tak, aby kazda sekce mela vlastni Dart test file.
- U writeable sekci v kazdem souboru overit minimalne create/read/update; kde docs umi i search, action nebo download, overit i tyto endpointy v tomtez souboru.
- Delete endpointy nevkladat nahodne do vsech souboru, ale pokryt je kompletne a explicitne v `test/live_cleanup_test.dart`.
- U read-only sekci overit list/detail/bootstrap flow vlastnim souborem.
- Zachovat request parity a model/error parity jako doplnkovou vrstvu pro edge cases, ktere nejsou vhodne pro live tenant.

### Faze 3a - Sekcni layout testu
- Vytvorit samostatne files po docs sekcich: `test/live_authorization_test.dart`, `test/live_account_test.dart`, `test/live_users_test.dart`, `test/live_bank_accounts_test.dart`, `test/live_number_formats_test.dart`, `test/live_subjects_test.dart`, `test/live_invoices_test.dart`, `test/live_invoice_payments_test.dart`, `test/live_invoice_messages_test.dart`, `test/live_expenses_test.dart`, `test/live_expense_payments_test.dart`, `test/live_inbox_files_test.dart`, `test/live_inventory_items_test.dart`, `test/live_inventory_moves_test.dart`, `test/live_generators_test.dart`, `test/live_recurring_generators_test.dart`, `test/live_todos_test.dart`, `test/live_events_test.dart`, `test/live_webhooks_test.dart`, plus `test/live_cleanup_test.dart`.
- Z `test/live_full_api_test.dart` presunout domenove scenare do techto sekcich souboru; tento soubor bud odstranit, nebo nechat jen jako legacy agregaci, pokud nebude duplicita zadouci.
- Vytvorit shared fixture layer v `test/support/` pro bootstrap klienta, lookup/create dependency grafu (subject -> invoice/expense -> payments -> attachments -> inventory item -> move -> generator -> recurring generator -> webhook -> inbox file).
- Zajistit, aby kazdy sekcni test file slo spustit samostatne proti prazdnemu tenantovi `testcompany1` a dovedl si vytvorit vlastni prerequisites nebo je deterministicky dohledat pres test prefixes/tags.

### Faze 4 - Verifikace
- Spustit `dart format .`, `dart analyze`, `flutter test`.
- Spustit live suite explicitne proti `FAKTUROID_SLUG=testcompany1`.
- Overit samostatne vsechny sekcni soubory v preserve modu a cleanup suite spoustet jen separatne pri explicitnim kroku.
- Drzet credentials pouze runtime; nic z nich neukladat do git historie ani fixture souboru.

- Live vrstva:
  - pouzit tenant `testcompany1`

- Omezeni:
